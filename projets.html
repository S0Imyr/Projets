Projet API RESTful avec DRF (en cours)

Mission :

Une entreprise de développement souhaite avoir une application de suivi des bugs et problèmes de leurs projets en développement. L'application doit être disponible pour les trois plateforme site web, Android et iOS. Les trois exploiteront les endpoints d'une API.

La mission est donc de réaliser cette API en utilisant Django REST Framework.

Elle doit tout d'abord permettre l'authentification des utilisateurs avec JWT. En l'absence d'authentification, aucun endpoint n'est accessible.

Les utilisateurs doivent pouvoir créer et avoir la liste des projets. Il faut ensuite être un contributeur du projet pour le lire. Pour le mettre à jour ou le supprimer, il faut en être l'auteur.

Les contributeurs d'un projet peuvent attribuer des problèmes à ce projet ou lire la liste des problèmes d'un projet. Pour mettre à jour ou supprimer un problème, il faut en être l'auteur.

Enfin les contributeurs d'un projet peuvent créer et lire des commentaires relatifs à un problème. La mise à jour ou la suppression d'un commentaire demande que l'utilisateur soit l'auteur.

Détails Repository

Compétences

  • Django Rest Framework
  • Documentation d'une API avec Postman
  • Introduction à l'OWASP

Apports

Le projet permet de parcourir les différentes possibilités pour créer des endpoints : ViewSet, APIView ou les GenericViewSet.

Il permet une mise en pratique des serialiseurs et de l'authentification.

Un autre point important est la gestion des permissions à différents niveaux.

Projet POO Python

Mission :

Il s'agit d'élaborer un programme pour un club d'échecs. Lors de tournoi d'échecs, le programme doit permettre d'établir quels joueurs vont s'affronter. Le programme doit respecter pour cela, les règles du tournoi suisse. Les tournois pouvant être interrompu et repris le lendemain, il doit être possible d'interrompre le tournoi de le reprendre au même stade. Enfin le programme doit permettre d'écrire un certain nombre de rapports :

  • la liste des joueurs ayant participé aux différents tournois
  • la liste des tournois
  • pour un tournoi, la liste des joueurs
  • pour un tournoi, la liste des matchs
  • pour un tournoi, la liste des tours
Pour les listes de joueurs, elles doivent être disponible par ordre alphabétique ou selon leur classement.

En termes de spécifications techniques, il est exigé de suivre un design pattern, il s'agit de se conformer à un modèle de conception MVC (Modèle, Vue, Contrôleur). Pour la base de donnée, on utilisera le package TinyDB. Enfin le code Python devra respecter la PEP8.

Détails Repository

Compétences

  • POO sous Python
  • Respecter un Design pattern
  • PEP8

Apports

Le projet est difficile à débuter puisque l'on part de rien mais est très intéressant dans la conception du modèle. La gestion des sauvegardes est aussi assez complexe puisqu'il est demandé de pouvoir arrêter et reprendre un tournoi.

Divers

Pour les contrôleurs, j'ai construit un contrôleur principal qui boucle entre les autres contrôleurs. La boucle prenant fin quand on appelle le contrôleur pour quitter.

Les règles du tournoi suisse dans l'élaboration des matchs ne sont pas entièrement respectées. Pour totalement s'y conformer, l'algorithme peut devenir complexe et cela n'était pas l'objectif du projet. J'y reviendrai certainement pour améliorer cela.

Actuellement, lorsque les matchs sont déterminés, si tous les joueurs restant se sont déjà rencontrés, les rencontres déjà fixées ne sont pas remises en cause et donc des joueurs peuvent se rencontrer deux fois.
